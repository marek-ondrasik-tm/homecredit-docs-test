## Začínáme

Tento online dokument popisuje API Home Credit pro partnery - prostředek, pomocí kterého mohou partneři svým zákazníkům nabídnout výhody jedné z platebních metod nebo produktů Home Credit.

Popisované služby mají za cíl umožnit zákazníkům hladký průběh procesu online nákupu nebo sjednání úvěru. Služby, které umožňují dosažení tohoto cíle, jsou rozděleny do následujících skupin:

| Skupina služeb                                           | Účel                                                                                                                                                      |
| ---                                                         | ---                                                                                                                                  |
| [Bezpečnost](./security)                             | Bezpečnostní pravidla a zásady                                                                                                                                |
| [Aplikační zdroje](./application-resources)   | Zdroje, které umožňují zpracování *aplikace* během celého jejího životního cyklu                                                                                |
| [Zdroje obchodního webu](./merchantsite-resources) | Zdroje, které by měl partner vyvinout a zpřístupnit, aby mu HC mohla asynchronně sdělovat důležité změny týkající se konkrétních *aplikací* |

### Základní implementace

Základním a minimálním způsobem implementace HCO API je implementovat alespoň:

1. [Přihlášení](./login-partner)
- Partner musí být autorizován, aby měl přístup k většině zdrojů
2. [Vytvoření aplikace](./create-application)
- Partner volá tento zdroj, aby zahájil proces schvalování úvěru vytvořením `aplikace`. `Aplikace` sama o sobě a její ID jsou nezbytné pro úspěšné dokončení procesu.
3. Přesměrování handlerů.
- Speciální URL adresy pro zákazníky, které zobrazují úspěšnost nebo zamítnutí platby (více informací [zde](#parameters-added-to-return-urls)).
4. [Oznámení o žádosti](./application-notification)
- Tento koncový bod musí být na vaší straně zpřístupněn, aby vám backend HC mohl asynchronně oznamovat důležité změny týkající se konkrétních žádostí.
5. [Označit položky objednávky jako odeslané/doručené](./mark-order-as-shipped)
- Home Credit nezačne proces finanční kompenzace, dokud není `objednávka` označena jako `ODESLÁNA` nebo `DORUČENA` .

## Zásady dokumentace

- atributy v objektu žádosti/odpovědi jsou volitelné, pokud není uvedeno jinak (příznak "povinné" pod názvem atributu)

- povinný atribut v nepovinném objektu znamená, že pokud je nepovinný objekt specifikován, musí obsahovat povinný atribut.

- všechny hodnoty v atributech požadavku/odpovědi jsou pouze příklady, s výjimkou hodnot výčtu - ty jsou jedinými možnými hodnotami pro daný atribut.

- dokumentace se řídí [sémantickým verzováním](https://semver.org/spec/v2.0.0.html), které rozlišuje mezi *hlavními*, *vedlejšími* a *opravnými* verzemi.

### Konvence pojmenování

- pro všechny názvy objektů a atributů používáme camelCase

- pro hodnoty výčtu používáme CAPITAL_UNDERSCORE

- v názvech zdrojů používáme množné number

### Klíčová slova použitá v tomto dokumentu (RFC 2119)

Klíčová slova "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED",  "MAY" a "OPTIONAL" v tomto dokumentu se interpretují tak, jak je popsáno v [RFC 2119](https://www.ietf.org/rfc/rfc2119.txt).

### Limity volání API

Když partner dosáhne limitu volání API (partner volá API vícekrát, než je jeho kvóta), obdrží HTTP chybu 429. K informování partnera o limitech používáme následující hlavičky odpovědi:

- `X-Rate-Limit-Limit` - Počet povolených požadavků v aktuálním období

- `X-Rate-Limit-Remaining` - počet zbývajících požadavků v aktuálním období

- `X-Rate-Limit-Reset` - počet sekund zbývajících v aktuálním období

### Verze

V URL používáme verzi API (např. `https://api.homecredit.cz/financing/v3/applications`). *Drobné* změny a *opravy* (viz níže), které nenarušují zpětnou kompatibilitu, NEZVYŠUJÍ verzi API, tj. MOHOU se objevit bez předchozího upozornění a vaše aplikace by měla být připravena je zpracovat.

*Drobné* změny zahrnují:

- přidání nového zdroje

- přidání nového volitelného parametru záhlaví/URL nebo volitelného atributu těla k požadavku

- přidání nového atributu k tělu odpovědi

- přidání nových chybových kódů a zpráv, za předpokladu, že struktura chyb je stejná

### Kódování odpovědi

Pokud není uvedeno jinak, všechny odpovědi jsou odesílány jako `Content-Type: application/json; charset=utf-8`.

### Kódy stavu HTTP

V celé API používáme následující kódy stavu, s výjimkou toku OAuth, kdy jsou kódy odpovědí předepsány v RFC

- 200 `OK` - požadavek byl úspěšný

- 201 `Created` - požadavek byl úspěšný a zdroj byl vytvořen

- 202 `Accepted` - požadavek byl přijat ke zpracování, ale zpracování nebylo dokončeno

- 204 `No Content` - přijali jsme požadavek partnera, ale není co vrátit (např.např. odpověď je prázdná)

- 400 "Bad Request" - syntaktická chyba, např. v požadavku chybí požadované parametry/atributy nebo hodnoty parametrů jsou nesprávného typu

- 401 "Unauthorized" - partner není autorizován

- 403 "Forbidden" - přístup odepřen (např. uživatel/aplikace nemá povoleno používat zdroj)

- 404 "Not Found" - zdroj nebyl nalezen

- 405 "Method Not Allowed" - zadaná metoda není pro zdroj povolena

- 422 "Unprocessable Entity" -  obchodní (sémantické) chyby. Žádost je správně formátovaná, ale nelze ji zpracovat (např. datum splatnosti platby je v minulosti). Chyby jsou uvedeny v těle odpovědi (viz níže)

- 429 `Too many requests` - partner překročil limit rychlosti (viz část [Limit volání API](#api-calls-limits) výše)

- 500 `Internal server error` - na naší straně došlo k chybě

- 503 `Service Unavailable` - je plánované přerušení služby

### Zpracování chyb

Kromě stavových kódů HTTP, které jsou hlavním indikátorem toho, že došlo k chybě, používáme také objekt `errors` k hlášení podrobnějších informací o chybách.

Příklad objektu errors:

```javascript
{
    ...

    errors: [
        {
            "code": "ERR_100",
            "message": "Neplatné number smlouvy",
            "severity": "ERROR",
            "attribute": "partyAccount.accountNumber",          // volitelné
            "ticketId": "UAT1:AMS:20160516-091658.450:45e4" // volitelné
        },
        {
            "code": 352,
            "message": "Nedostatečné prostředky pro realizaci platebního příkazu",
            "severity": "WARN"
        },
        {
            "code": 523,
            "message": "Tento příkaz spustí operaci směny měny",
            "severity": "INFO"
        }
    ]
}
```

Atributy objektu chyby
| Název atributu | Popis                                                     |
| ---            | ---                                                             |
| code           | jedinečný kód chyby                                               |
| message        | popis chyby čitelný pro člověka (nelokalizovaný)                |
| severity       | závažnost chyby (viz níže)                   |
| atribut      | json cesta atributu požadavku, který způsobil chybu (volitelné) |
| ticketId       | interní ID tiketu, používané pro zpětné sledování chyby                 |

Existují 3 úrovně závažnosti chyby:

- `ERROR` - kritická chyba, provedení nelze pokračovat. To MUSÍ být také označeno příslušným stavovým kódem HTTP (např. `422 Unprocessable Entity`)

- `WARN` - nekritická chyba, provedení lze pokračovat, ale je doporučeno další uživatelské zásahy (aby požadavek mohl pokračovat, partner MUSÍ specifikovat tento chybový kód v atributu požadavku `override`). To MŮŽE být také označeno příslušným stavovým kódem HTTP.

- `INFO` - pouze informace, provádění může pokračovat bez interakce uživatele.

### Formáty

- **date** a **time** používají formátování [ISO 8601](https://en.wikipedia.org/wiki/ISO_8601), např.:
  - datum je vyjádřeno jako `YYYY-mm-dd`. Časové pásmo (místní) je přidáno v případě potřeby.
  - čas je vyjádřen jako `Thh:mm:ss`. Časové pásmo se přidává v případě potřeby. Akceptuje se také čas s libovolným počtem číslic představujících milisekundy (pokud jsou splněna kritéria formátu [ISO 8601](https://en.wikipedia.org/wiki/ISO_8601)), ale vždy se vrací čas bez milisekund.
  - den v týdnu je vyjádřen číslem 1..7, přičemž 1 je pondělí.
  - týden č. 1 je týden, ve kterém je první čtvrtek v roce.

- **telefonní čísla** používají mezinárodní formát začínající znakem "+" a obsahující kód země (příklad platného čísla: `+420739111222`)

- **formát čísel** formát čísel je definován [standardem JSON](http://www.json.org), např. desetinná místa jsou oddělena znakem `.`

- **formát peněz** používá formátování [ISO 4217](https://en.wikipedia.org/wiki/ISO_4217) (v menších jednotkách, např.: 12590 v menších jednotkách představuje 125,90 CZK/EUR)

## Parametry přidané do návratových URL

- Během přesměrování zákazníka z front-endu Home Credit zpět na web partnera (na jednu z poskytnutých URL - `approvedRedirectUrl`, `rejectedRedirectUrl`) jsou přidány některé potenciálně užitečné informace o výsledku zákazníka ve front-endu Home Credit:

- `applicationId` (string) - Identifikátor žádosti HCO

- `applicationStatus` (enum) - Stav žádosti

- `orderNumber` (string) - Interní number objednávky e-shopuv rámci volání API [Vytvořit žádost](./create-application) (je-li k dispozici)

- `orderStatus` (enum) - stav objednávky (je-li k dispozici)

- `cashStatus` (enum) - stav hotovostní půjčky (je-li k dispozici)

- `withdrawal` (boolean) - True, pokud je nákup zákazníka financován přímo z jeho již existujícího revolvingového úvěru; false, pokud je vytvářena nová smlouva (je-li k dispozici)

- `downPayment` (number) - Hodnota zálohy, obvykle nastavena na 0 (používá se formát menších jednotek - 100000 představuje 1000, [ISO 4217](https://en.wikipedia.org/wiki/ISO_4217))

- `checkSum` (string) - hashovaná zpráva

- např. `http://youreshop.cz?&amp;applicationId;=123e4567-e89b-12d3-a456-426655440000&amp;applicationStatus;=CONFIRMED&amp;orderNumber;=018884&amp;orderStatus;=TO_SHIP&amp;downPayment;=100000&amp;checkSum;=443E150D3406F6A8DC62C1DB224AFDD84FA9907071BB1AD9B2C7701031793662191501B26AADF3A31361341EA176C80B23D8A9A5E5B5219538289945FE0C20A7`

## Průvodce migrací: API v1 → v3

Tento průvodce migrací se zaměřuje na změny v sekci [Aplikační zdroje](./application-resources) při přechodu z API verze 1 na verzi 3. Cílem je poskytnout přehled klíčových změn, které vyžadují úpravy implementace.

### 1. Přehled změn koncových bodů API

#### 1.1 Struktura URL

- **v1:** `/financing/v1/applications`

- **v3:** `/financing/v3/applications`

**Ověřování:** Obě verze API používají stejný ověřovací koncový bod `/authentication/v1/partner` k získání bezpečnostního tokenu (viz část [Přihlášení partnera](./login-partner)).

#### 1.2 Nové koncové body ve verzi v3

- `GET /financing/v3/applications/{applicationId}` - načtení podrobností aplikace

- `PUT /financing/v3/applications/{applicationId}/cancel` - zrušení aplikace

- `PUT /financing/v3/applications/{applicationId}/orders/ship` - označení objednávky jako odeslané

- `PUT /financing/v3/applications/{applicationId}/orders/deliver` - označení objednávky jako doručené

#### 1.3 Změněné koncové body z v1

- Koncový bod zrušení byl upraven (viz část [2.4](#24-applicationresponse---major-changes))

### 2. Změny datové struktury

#### 2.1 ApplicationRequest - významné strukturální změny

V1 podporovala pouze fyzické osoby. V3 zavádí nové rozlišení typů zákazníků:

- **Fyzická osoba** (`PERSON`) - běžný zákazník

- **Osoba samostatně výdělečně činná** (`ENTREPRENEUR`) - živnostník

- **Právnická osoba** (`LEGAL_ENTITY`) - společnost, LLC, korporace atd.

### 3. Změny v životním cyklu

#### 3.1 Vytvoření aplikace

**Dostupnost:** Žádná změna v obou verzích  
**Metoda HTTP:** `POST`  
**Dopad změny:** **KRITICKÝ**

**Klíčové změny:**
1. Zcela odlišná struktura požadavku (viz sekce [2.1](#21-applicationrequest---major-structural-changes) - [2.3](#23-order---major-changes))
2. Změna struktury odpovědi
3. Duplicitní logika zůstává stejná (200 vs 201 vs 422)

#### 3.2 Podrobnosti aplikace

| Operace | v1 | v3 | Změna |
|---------|----|----|-------|
| **Získat podrobnosti o žádosti** | `GET /financing/v1/applications/{applicationId}` | `GET /financing/v3/applications/{applicationId}` | **NOVÝ koncový bod ve v3** - doporučeno implementovat |
| **Získat objednávku** | `GET /financing/v1/orders?applicationId={id}` | Odstraněno | Podrobnosti objednávky nyní prostřednictvím podrobností aplikace |

**Dopad:** **STŘEDNÍ** - změna způsobu získávání informací o objednávce

#### 3.3 Označit zásilku/dodávku

| Operace | v1 | v3 | Dopad |
|---------|----|----|-------|
| **Označit jako odeslané** | `PUT /financing/v1/applications/{id}/order/send` | `PUT /financing/v3/applications/{id}/orders/ship` | **NÍZKÝ** - změna URL |
| **Označit jako doručené** | `PUT /financing/v1/applications/{id}/order/deliver` | `PUT /financing/v3/applications/{id}/orders/deliver` | **NÍZKÝ** - změna URL |

**Klíčová změna:** Ve verzi v3 neexistuje koncový bod pro `send`, pouze `ship` a `deliver`.

**Dostupnost ve verzi v3:**

- **Odeslat:** Pouze pro stavy `CONFIRMED` + `TO_SHIP`

- **Doručit:** Pouze pro stavy `CONFIRMED` + `TO_SHIP`

**Odpověď ve verzi v3:** Stav 204 (bez obsahu) namísto 200 s tělem odpovědi

#### 3.4 Zrušení žádosti

| Operace | v1 | v3 | Dopad |
|---------|----|----|-------|
| **Zrušení žádosti** | `PUT /financing/v1/applications/{id}/cancel` | `PUT /financing/v3/applications/{id}/cancel` | **NÍZKÝ** |

**Klíčové změny:**
1. Mírně upravené tělo požadavku (v zásadě stejné hodnoty výčtu `reason`)
2. Odpověď ve v3: Stav 200 (namísto 200 s objektem Application)
3. Dostupnost ve v3 je rozšířena (lze zrušit více stavů)

**Nové stavy dostupné pro zrušení ve v3:**

- `CREATED` + `IN_PROCESS`

- `PRESCORED` + `IN_PROCESS`

- `ALTERNATIVE` + `IN_PROCESS`

- `APPROVED` + `IN_PROCESS`

- `CONFIRMED` + všechny kromě `PAID`

- `CANCELLED` + `CANCELLED` (idempotentní)

#### 3.5 Úprava objednávky

**v1:** `PUT /financing/v1/applications/{applicationId}/order`  
**v3:** **Odstraněno**

**Dopad:** **VYSOKÝ** - pokud byl tento koncový bod používán ke změně čísla objednávky nebo symbolu proměnné, je nutné najít alternativní řešení.

### 4. Nové funkce ve verzi v3

#### 4.1 Podpora zprostředkovaných objednávek

Verze v3 zavádí možnost vytvářet "zprostředkované" objednávky pro obchodní vztahy mezi více stranami.

**Příklad použití:**

```json

{
  "financedSubject": {
    "type": "ORDER",
    "order": {
      "type": "BROKERED",
      "brokered": {
        "merchantOrders": [
          {
            "name": "MALL",
            "number": "22222222",
            "items": [...]
          }
        ]
      }
    }
  }
}

```

#### 4.2 Podpora podnikatelů a právnických osob

Ve verzi v3 lze nyní vytvářet aplikace pro:

- `PERSON` (fyzická osoba) - odpovídá verzi v1

- `ENTREPRENEUR` (podnikatel)

- `LEGAL_ENTITY` (právnická osoba)

#### 4.3 Kroky pracovního postupu

Odpověď v3 obsahuje `workflowSteps[]` - strukturované informace o dalších krocích procesu.

#### 4.4 Kontext vzniku

Nová struktura pro sledování zdroje žádosti:

- `precheckIds` - odkaz na předběžnou kontrolu

- `sourceId` - identifikátor kampaně

- `extendedData` - parametry UTM, informace o sledování
