## Getting started

This online document describes Home Credit API for partners - the means for partners through which they can offer their customers the benefits of one of Home Credit's payment methods or products.

Described services aim to allow customers the smooth running through the process of an online purchase or loan arrangement. The services that allow achieving this goal are divided into the following groups:

| Group of services                                           | Purpose                                                                                                                                                      |
| ---                                                         | ---                                                                                                                                                          |
| [Security](./security)                             | Security rules and principles                                                                                                                                |
| [Application resources](./application-resources)   | Resources that allows processing of *application* during its whole life cycle                                                                                |
| [Merchantsite resources](./merchantsite-resources) | Resources that should be developed and exposed by a partner to allow HC asynchronously inform it about important changes regarding particular *applications* |

### Basic implementation

The basic and minimum way how to implement HCO API is to at least implement:
1. [Login](./login-partner)
    - Partner must be authorized to be able to access the most resources
2. [Create application](./create-application)
    - Partner calls this resource to start the loan approval process by creating an `application`. `Application` itself and its ID are essential for successful process completion.
3. Redirect handlers.
    - Special URLs for customers that shows payment success or rejection (more info [here](#parameters-added-to-return-urls)).
4. [Application notification](./application-notification)
    - This endpoint has to be exposed on your side to allow HC back-end to asynchronously notify you about the important changes regarding particular applications.
5. [Mark order items as shipped/delivered](./mark-order-as-shipped)
    - Home Credit does not start the process of financial compensation unless the `order` is marked as `SHIPPED` or `DELIVERED` .

It is necessary to fill in all attributes marked as 'required' in the request. Filling optional parameters will result in a better application approval rate. Partner should always consider filling as many attributes as possible.

## Documentation principles

- attributes in request/response object are optional unless stated otherwise (`required` flag under attribute name)

- a required attribute in an optional object means, that if an optional object is specified, it must contain the required attribute.

- all values in request/response attributes are just examples, except for enum values - these are the only possible values for the given attribute.

- documentation adheres to [semantic versioning](https://semver.org/spec/v2.0.0.html), which makes a distinction between *major*, *minor*, and *patch* versions.

### Naming conventions

- we use camelCase for all object and attribute names

- we use CAPITAL_UNDERSCORE for enum values

- we use plural in resource names

### Key words used in this document (RFC 2119)

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED",  "MAY", and "OPTIONAL" in this document are to be interpreted as described in [RFC 2119](https://www.ietf.org/rfc/rfc2119.txt).

### API calls limits

When partner reaches API calls limit (partner calls the API more times than is partner's quota), he receives HTTP error 429. To inform partner about limits we use following response headers:

- `X-Rate-Limit-Limit` - The number of allowed requests in the current period

- `X-Rate-Limit-Remaining` - The number of remaining requests in the current period

- `X-Rate-Limit-Reset` - The number of seconds left in the current period

### Versioning

We use API version in URL (e.g. `https://api.homecredit.cz/financing/v3/applications`). *Minor* changes and *patches* (see below) that don't break backwards compatibility do NOT increase API version, e.g. they MAY happen without prior notice and your application should be ready to handle them.

*Minor* changes include:

- adding new resource

- adding new optional header/URL parameter or optional body attribute to request

- adding new attribute to response body

- adding new error codes and messages, provided that error structure is the same

### Response encoding

Unless stated otherwise, all responses are sent as `Content-Type: application/json; charset=utf-8`

### HTTP status codes

We use following status codes throughout the API, except for OAuth flow when response codes are prescribed in RFC

- 200 `OK` - request was successful

- 201 `Created` - request was successful and resource was created

- 202 `Accepted` - request has been accepted for processing, but the processing has not been completed

- 204 `No Content` - we accepted partner's request but there is nothing to return (e.g. response is empty)

- 400 `Bad Request` - syntax error, e.g. request is missing required parameters/attributes or parameter values are of incorrect type

- 401 `Unauthorized` - partner is not authorized

- 403 `Forbidden` - access denied (e.g. user / application is not allowed to use the resource)

- 404 `Not Found` - resource could not be found

- 405 `Method Not Allowed` - specified method is not allowed for resource

- 422 `Unprocessable Entity` -  business (semantic) errors. Request is well-formed, but cannot be processed (e.g. payment due date is in past). Errors are specified in response body (see below)

- 429 `Too Many Requests` - partner exceeded the rate limit (see section [API calls limit](#api-calls-limits) above)

- 500 `Internal Server Error` - something went wrong on our side

- 503 `Service Unavailable` - there is planned service outage

### Error handling

Besides HTTP status codes, which are the main indication if something goes wrong, we also use `errors` object to report more details about errors.

Errors object example:

```javascript
{
    ...

    errors: [
        {
            "code": "ERR_100",
            "message": "Invalid contract number",
            "severity": "ERROR",
            "attribute": "partyAccount.accountNumber",          // optional
            "ticketId": "UAT1:AMS:20160516-091658.450:45e4" // optional
        },
        {
            "code": 352,
            "message": "Insufficiend funds for payment order realization",
            "severity": "WARN"
        },
        {
            "code": 523,
            "message": "This order will trigger currency exchange operation",
            "severity": "INFO"
        }
    ]
}
```

Error object attributes
| Attribute name | Description                                                     |
| ---            | ---                                                             |
| code           | unique error code                                               |
| message        | human readable error description (non-localized)                |
| severity       | error severity (see below)                   |
| attribute      | json path of request attribute that caused the error (optional) |
| ticketId       | internal ticket ID, used for error backtracking                 |

There are 3 levels of error severity:

- `ERROR` - critical error, execution cannot continue. This MUST be indicated also by appropriate HTTP status code (e.g. `422 Unprocessable Entity`)

- `WARN` - non-critical error, execution can continue but further user interaction is advisable (for request to proceed, partner MUST specify this error code in `override` request attribute). This MIGHT be indicated also by appropriate HTTP status code.

- `INFO` - information only, execution can continue without user interaction.

#### Errors overview

| Status code | Code                  | Explanation                                                                     | Message                                                                                                        |
| ---         | ---                   | ---                                                                             | ---                                                                                                            |
| 400         | INVALID_REQUEST       | Request was not well formatted (malformed request syntax, size too large, etc.) | (*Various messages possible*) e.g. Unsupported combination of application status and order status              |
| 404         | OBJECT_NOT_FOUND      | The requested resource could not be found                                       | (*Various messages possible*) e.g. Specified object was not found                                              |
| 422         | NOT_ALLOWED           | Operation is not allowed (internal reason)                                      | (*Various messages possible*) e.g. Not allowed (other reason)                                                  |
| 500         | INTERNAL_SERVER_ERROR | Unexpected condition was encountered and no more specific message is suitable   | (*Various messages possible*) e.g. Failed to parse given JSON                                                  |

### Formats

- **date** and **time** uses [ISO 8601](https://en.wikipedia.org/wiki/ISO_8601) formatting, e.g.:
  - date is represented as `YYYY-mm-dd`. Timezone (local) is added when necessary.
  - time is represented as `Thh:mm:ss`. Timezone is added when necessary. Time with arbitrary number of digits respresenting miliseconds is also accepted (when [ISO 8601](https://en.wikipedia.org/wiki/ISO_8601) format criteria is met) but the time without miliseconds is always returned.
  - day of week is represented as number 1..7, with 1 being Monday
  - week no. 1 is the week with the year's first Thursday in it

- **phone numbers** uses international format starting with '+' and including country code (example of valid number: `+420739111222`)

- **numbers format** number format is defined by [JSON standard](http://www.json.org), e.g. decimals are separated by `.`

- **money format** uses [ISO 4217](https://en.wikipedia.org/wiki/ISO_4217) formatting (in minor units, e.g.: 12590 in minor units represents 125,90 CZK/EUR)

## Parameters added to return URLs

- During redirection of the customer from Home Credit's front-end back to partner's website (to one of the provided URLs - `approvedRedirectUrl`, `rejectedRedirectUrl`) there are some potentially useful information about a customer's outcome in Home Credit's front-end added:

- `applicationId` (string) - HCO application identifier

- `applicationStatus` (enum) – Application status

- `orderNumber` (string) - E-shop's internal order number provided within API call of [Create application](./create-application) (if available)

- `orderStatus` (enum) - Order status (if available)

- `cashStatus` (enum) - Cash loan status (if available)

- `withdrawal` (boolean) – True when the customer's purchase is financed directly from his already existing revolving loan; false when a new contract is being created (if available)

- `downPayment` (number) – Value of down payment, usually set to 0 (minor units format is used - 100000 represents 1000, [ISO 4217](https://en.wikipedia.org/wiki/ISO_4217))

- `checkSum` (string) – hashed message

- e.g. `http://youreshop.cz?&applicationId=123e4567-e89b-12d3-a456-426655440000&applicationStatus=CONFIRMED&orderNumber=018884&orderStatus=TO_SHIP&downPayment=100000&checkSum=443E150D3406F6A8DC62C1DB224AFDD84FA9907071BB1AD9B2C7701031793662191501B26AADF3A31361341EA176C80B23D8A9A5E5B5219538289945FE0C20A7`

## Migration Guide: API v1 → v3

This migration guide focuses on changes in the [Application resources](./application-resources) section when transitioning from API version 1 to version 3. The goal is to provide an overview of key changes that require implementation adjustments.

### 1. API endpoint changes overview

#### 1.1 URL structure

- **v1:** `/financing/v1/applications`

- **v3:** `/financing/v3/applications`

**Authentication:** Both API versions use the same authentication endpoint `/authentication/v1/partner` to obtain a security token (see section [Login partner](./login-partner)).

#### 1.2 New endpoints in v3

- `GET /financing/v3/applications/{applicationId}` - retrieve application detail

- `PUT /financing/v3/applications/{applicationId}/cancel` - cancel application

- `PUT /financing/v3/applications/{applicationId}/orders/ship` - mark order as shipped

- `PUT /financing/v3/applications/{applicationId}/orders/deliver` - mark order as delivered

#### 1.3 Changed endpoints from v1

- Cancel endpoint has been modified (see section [2.4](#24-applicationresponse---major-changes))

### 2. Data structure changes

V1 only supported natural persons. V3 introduces new customer type distinction:

- **Natural person** (`PERSON`) - regular customer

- **Self-employed person** (`ENTREPRENEUR`) - sole proprietor

#### 2.2 Customer - major changes

| Change | v1 | v3 | Impact |
|-------|----|----|-------|
| **Customer structure** | Direct attributes (`firstName`, `lastName`, ...) | Typed structures by customer type (`person`, `entrepreneur`, `legalEntity`) | **HIGH** - Refactoring of all calls required |
| **Customer requirement** | `required` (with optional attributes possible) | `required` with `type`, but data may be `optional` for `ONLINE` process | **MEDIUM** |
| **Email and phone** | Direct attributes | `contacts[]` array with contact type | **HIGH** |
| **Addresses** | `addresses[].addressType` enum | `addresses[].flags[]` array | **MEDIUM** |
| **IP address** | `customer.ipAddress` | `customer.person.ipAddress` / `customer.entrepreneur.ipAddress` / `customer.legalEntity.ipAddress` | **LOW** |
| **Person identification** | Not in base structure | `customer.person.identificationNumber` | **LOW** - new field |

| Change | v1 | v3 | Impact |
|-------|----|----|-------|
| **Order location** | Root level `order` | Nested in `financedSubject.order` | **HIGH** |
| **Order type** | Not present | `type`: `STANDARD` / `BROKERED` | **HIGH** - new required structure |

| **MerchantUrls** | Root level `merchantUrls` | Moved to `originationContext.channel.online` | **HIGH** |
| **Variable symbols** | `variableSymbols[]` array | `variableSymbol` string (single) + `specificSymbol` | **MEDIUM** |

```json
        "approvedRedirect": "https://...",
        "rejectedRedirect": "https://...",
        "notificationEndpoint": "https://..."
      }
    }
  }
}
```

| **Order state** | Directly in response | Only via `GET /applications/{id}` | **MEDIUM** |

### 3. Lifecycle operation changes

#### 3.1 Create application

**Availability:** No change in both versions  
**HTTP method:** `POST`  
**Change impact:** **CRITICAL**

**Key changes:**
1. Completely different request structure (see sections [2.1](#21-applicationrequest---major-structural-changes) - [2.3](#23-order---major-changes))
2. Response structure change
3. Duplicate logic remains the same (200 vs 201 vs 422)

#### 3.2 Application detail

| Operation | v1 | v3 | Change |
|---------|----|----|-------|
| **Get Application Detail** | `GET /financing/v1/applications/{applicationId}` | `GET /financing/v3/applications/{applicationId}` | **NEW endpoint in v3** - recommended to implement |
| **Get Order** | `GET /financing/v1/orders?applicationId={id}` | Removed | Order detail now via application detail |

**Impact:** **MEDIUM** - change in how to retrieve order information

#### 3.3 Mark shipment/delivery

| Operation | v1 | v3 | Impact |
|---------|----|----|-------|
| **Mark as Sent** | `PUT /financing/v1/applications/{id}/order/send` | `PUT /financing/v3/applications/{id}/orders/ship` | **LOW** - URL change |
| **Mark as Delivered** | `PUT /financing/v1/applications/{id}/order/deliver` | `PUT /financing/v3/applications/{id}/orders/deliver` | **LOW** - URL change |

**Key change:** In v3, there is no endpoint for `send`, only `ship` and `deliver`.

**Availability in v3:**

- **Ship:** Only for `CONFIRMED` + `TO_SHIP` states

- **Deliver:** Only for `CONFIRMED` + `TO_SHIP` states

**Response in v3:** Status 204 (No Content) instead of 200 with response body

#### 3.4 Cancel application

| Operation | v1 | v3 | Impact |
|---------|----|----|-------|
| **Cancel Application** | `PUT /financing/v1/applications/{id}/cancel` | `PUT /financing/v3/applications/{id}/cancel` | **LOW** |

**Key changes:**
1. Request body slightly modified (same `reason` enum values in principle)
2. Response in v3: Status 200 (instead of 200 with Application object)
3. Availability in v3 is extended (more states can be cancelled)

**New states available for cancel in v3:**

- `CREATED` + `IN_PROCESS`

- `PRESCORED` + `IN_PROCESS`

- `ALTERNATIVE` + `IN_PROCESS`

- `APPROVED` + `IN_PROCESS`

- `CONFIRMED` + any except `PAID`

- `CANCELLED` + `CANCELLED` (idempotent)

#### 3.5 Order modification

**v1:** `PUT /financing/v1/applications/{applicationId}/order`  
**v3:** **Removed**

**Impact:** **HIGH** - if this endpoint was used for changing order number or variable symbol, an alternative solution must be found

### 4. New features in v3

#### 4.2 Entrepreneurs and legal entities support

In v3, applications can now be created for:

- `ENTREPRENEUR` (entrepreneur)

- `LEGAL_ENTITY` (legal entity)

The v3 response contains `workflowSteps[]` - structured information about next process steps.
